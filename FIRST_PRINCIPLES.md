# First-Principles Analysis — What the Problem Actually Asks

Strip everything back to the problem statement. No multi-agent hype. No theoretical edges. Just: what's asked, what we have, what we need to build.

---

## The 3 Deliverables (from ps.md)

### D1: Automated flare database (nowcast)
> "An automated database of nowcasted solar flares, generated by combining soft and hard X-ray light curves."

**What this means:** Run SoLEXS + HEL1OS light curves through a detection algorithm. Output: CSV/database with start_time, peak_time, end_time, class for every flare detected.

**What we have:**
- 9 days with both instruments, containing **104 GOES-cataloged flares** (51 C, 44 M, 9 X)
- 5 additional SoLEXS-only days with 49 more flares
- GOES catalog as ground truth to validate against

**What we need to build:**
1. FITS reader for both instruments (SoLEXS SDD2 `.lc` + HEL1OS CdTe/CZT `lightcurve_*.fits`)
2. Flare detection algorithm on each instrument independently
3. Catalogue merger that combines both detections
4. Validation against GOES catalog

### D2: Predictive model with lead time
> "A trained model capable of forecasting flares with a quantifiable lead time."

**What this means:** Before a flare peaks, the system should trigger an alert. Scored on: how many minutes before the peak?

**What we have:**
- Measured onset-to-GOES-peak lead times: 6.7 min (X1.0), 11.4 min (X6.9), 13.1 min (X4.4)
- CZT 40+ keV peaks ~2.5 min before CdTe 5-20 keV (extra lead from harder bands)
- Running integral of HXR predicts eventual SXR class before SXR peaks
- 9 X-class + 44 M-class events on dual-instrument days to train/validate

**What we need to build:**
1. Onset detection (CUSUM or threshold-based)
2. Multi-band trigger hierarchy (CZT 40+ keV first, CdTe second)
3. Class prediction via running integral
4. Lead time measurement + reporting per class

### D3: Visualization interface
> "Interface that visualizes the X-ray light curves and triggers with visual alerts when a flare is nowcasted or forecasted."

**What this means:** A dashboard showing live (or replay) light curves with alert indicators.

**What we need to build:**
1. Streamlit/Plotly app
2. Dual-panel: SoLEXS + HEL1OS light curves
3. Alert overlay (WATCH → WARNING → ALERT)
4. Flare catalog table

## The 3 Evaluation Criteria

### E1: Detection of low- and high-class flares
- **Low-class (A/B/C):** SoLEXS SDD2 only. HEL1OS can't see these (too few photons).
  - We have 51 C-class on dual-instrument days, 79 C-class total
  - No A/B-class in GOES catalog for our dates (below GOES sensitivity)
  - Detection method: threshold on SDD2 light curve + matched filter
- **High-class (M/X):** Both instruments.
  - 44 M-class + 9 X-class on dual-instrument days
  - Detection method: HEL1OS CUSUM + SoLEXS confirmation

### E2: High TPR + low False Alarm Rate
- **TPR** = what fraction of real flares we detect
- **FAR** = what fraction of our alerts are false
- Key: the AND-gate (both instruments must fire for M/X) crushes false alarms
- For C-class, single-instrument detection has higher FAR — accept it, report honestly

### E3: Lead time (minutes before flare peak)
- **Measured from our data:**
  - HXR onset (5σ) → GOES SXR peak: **6.7 to 13.1 min** (varies by event)
  - CZT 40+ keV peak → CdTe 5-20 keV peak: **~2.5 min** additional advantage
  - Peak-to-peak Neupert: **0 to 3.2 min** (unreliable for impulsive events)
- **Best framing:** onset-based lead, reported per class, with survival curves

## What Actually Matters (cutting through the noise)

The PLAN.md and UNDERSTANDING.md have ~500 lines of strategy. Here's what actually needs to happen:

### The 5 essential scripts:

```
1. ingest.py          — Load SoLEXS SDD2 LC + HEL1OS CdTe/CZT LC for any date
                         Handle: NaN gaps, 12-hour HEL1OS stitching, time alignment
                         Output: aligned DataFrames with columns [time, sxr_counts, hxr_cdte_5_20, hxr_czt_40_60, ...]

2. detect.py          — Flare detection on each instrument
                         SoLEXS: rolling percentile background + threshold crossing + peak finding
                         HEL1OS: multi-band CUSUM (CZT 40+ primary, CdTe secondary)
                         Output: per-instrument flare list [start, peak, end, peak_counts]

3. catalogue.py       — Merge detections + classify
                         Match SoLEXS + HEL1OS detections within ±2 min
                         Assign class from SoLEXS peak flux (calibrate against GOES)
                         AND-gate: both must fire for M/X classification
                         Output: master_catalogue.csv

4. forecast.py        — Onset detection + lead time measurement
                         CZT 40+ keV onset → earliest alert
                         Running integral → class prediction before SXR peak
                         Output: alerts with lead_time_minutes per event

5. app.py             — Streamlit dashboard
                         Replay a flare day, show SXR + HXR light curves
                         Alert overlay, catalog table
```

### The 3 validation outputs:

```
1. master_catalogue.csv     — Every detected flare with metadata
2. metrics_table.md         — TPR, FAR, TSS per class (C/M/X) vs GOES ground truth
3. lead_time_per_event.csv  — Lead time for each M/X event, with per-class summary
```
